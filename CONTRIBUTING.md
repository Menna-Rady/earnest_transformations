# Contributing

Thanks for contributing to Earnest Transformations! This guide covers how to set up your environment and the workflow expected for changes to this repository.

## Prerequisites

- Python 3.8+
- dbt-core and the appropriate adapter (dbt-snowflake)
- Access to Snowflake credentials (ask the maintainer; see `.env.example`)
- (For Spark work) Java/Spark runtime — see `spark/requirements.txt`

## Project Domains

This repository contains three distinct domains. Know which one you're touching:

| Domain | Folder | Stack |
|---|---|---|
| Scraping | `scraping/` | Python |
| Data Processing | `spark/` | PySpark, LLM agents |
| Data Modeling | `models/`, `macros/`, `seeds/`, `snapshots/` | dbt + Snowflake |

Each domain has its own `README.md` describing setup specifics.

## Branching & Workflow

1. Create a branch from `main` named `type/short-description` (e.g. `docs/architecture`, `fix/dim-seller-nulls`, `feat/discount-metric`).
2. Make your changes.
3. Run the relevant checks before opening a PR (see below).
4. Open a Pull Request against `main` with a clear description of *what* changed and *why*.
5. Wait for CI to pass (see `.github/workflows/dbt-ci.yml`) before requesting review.

## dbt Development Guidelines

- All new models must include corresponding schema tests in `schema.yml` (unique, not_null, relationships, accepted_values as appropriate).
- Document new columns and models with descriptions in `schema.yml`.
- Follow existing naming conventions: snake_case, `fact_*` for fact tables, `dim_*` for dimension tables, `stg_*` for staging models.
- If you add or change a measure in the fact table, update the corresponding semantic model and metrics in `models/marts/schema.yml`.
- Before committing, run:

```bash
dbt parse        # validate project compiles
dbt build        # run + test all models
dbt test         # run tests only
```

## Spark Development Guidelines

- New transformers go in `spark/transformers/`; keep each transformer focused on one responsibility.
- New custom errors should subclass the appropriate base in `spark/errors/`.
- Avoid committing large data artifacts (Parquet caches, CSVs) generated by local runs — check `.gitignore` covers your output paths before committing.

## Scraping Development Guidelines

- Document any new scraper target (site, schema of scraped fields, frequency) in `scraping/README.md`.
- Keep scraped output schemas consistent with what `spark/transformers/` expects as input — coordinate changes across both folders.
